# TIL (Today I Learned) – Unity에서 URP(Universal Render Pipeline)의 개념과 도입 시 주의점

## 주제 선택 이유

협업 프로젝트에서 URP 기반으로 프로젝트가 생성되어 있었지만,
기존 Built-in Render Pipeline과 무엇이 다른지 정확히 설명하기 어려웠다.
특히 셰이더, 라이팅, 포스트 프로세싱에서 차이가 발생하는 이유를 공식 문서 기준으로 정리할 필요가 있었다.

---

(A) 공식 문서 기준 팩트
1. URP란 무엇인가

URP(Universal Render Pipeline)는 Unity에서 제공하는 Scriptable Render Pipeline(SRP) 중 하나로,
모바일, 콘솔, PC 등 다양한 플랫폼에서 성능과 품질의 균형을 목표로 설계된 렌더 파이프라인이다.

Unity 공식 문서에서는 URP를 다음과 같이 설명한다.

Universal Render Pipeline is a prebuilt Scriptable Render Pipeline, optimized for performance and scalability across a wide range of platforms.

📎 출처: Unity Manual – Universal Render Pipeline
https://docs.unity3d.com/Manual/universal-render-pipeline.html

2. URP는 Built-in Render Pipeline과 구조적으로 다르다

URP는 기존의 Built-in Render Pipeline과 달리,
렌더링 과정이 Scriptable Render Pipeline(SRP) 기반으로 구현되어 있다.

이는 렌더링 흐름이 고정된 엔진 내부 코드에 의존하지 않고,
C# 스크립트로 렌더링 단계를 정의할 수 있는 구조임을 의미한다.

📎 출처: Unity Manual – Scriptable Render Pipeline
https://docs.unity3d.com/Manual/ScriptableRenderPipeline.html

3. URP에서는 전용 셰이더를 사용해야 한다

URP는 Built-in Render Pipeline용 셰이더(Standard Shader)를 지원하지 않는다.
URP 프로젝트에서는 **URP 전용 셰이더(Universal Lit, Simple Lit 등)**를 사용해야 한다.

Unity는 명시적으로 다음과 같이 안내한다.

Shaders written for the Built-in Render Pipeline are not compatible with URP.

📎 출처: Unity Manual – URP Shaders
https://docs.unity3d.com/Manual/urp-shaders.html

4. 라이팅과 포스트 프로세싱 방식이 다르다

URP는 자체적인 라이팅 시스템과 포스트 프로세싱 스택을 사용한다.
기존 Built-in Pipeline에서 사용하던 Post Processing Stack v2는 URP에서 사용되지 않는다.

URP에서는 Volume 시스템을 통해 후처리 효과를 제어한다.

📎 출처: Unity Manual – Volumes in URP
https://docs.unity3d.com/Manual/urp-volumes.html

5. 프로젝트 생성 이후 파이프라인 변경은 비용이 크다

Unity 공식 문서에서는 프로젝트 중간에 렌더 파이프라인을 변경할 경우
머티리얼, 셰이더, 라이팅 설정을 다시 설정해야 할 수 있음을 명시한다.

📎 출처: Unity Manual – Render Pipeline comparison
https://docs.unity3d.com/Manual/render-pipelines-feature-comparison.html

(B) 업계 관행

모바일 / 멀티 플랫폼 프로젝트에서는 URP를 기본 선택지로 사용하는 경우가 많다

협업 프로젝트에서는 초기 세팅 단계에서 URP 여부를 확정하는 것이 일반적이다

URP 사용 시:

커스텀 셰이더 사용 전, Shader Graph 활용을 우선 고려

라이팅 문제 발생 시 Built-in 방식으로 접근하지 않고 URP 문서를 먼저 확인

※ 위 내용은 공식 문서에 명시된 규칙이 아니라, 실무 및 팀 프로젝트에서 흔히 따르는 방식이다.

(C) 나의 해석

URP는 단순히 “가벼운 렌더링 파이프라인”이 아니라,
렌더링 구조 자체가 다르다는 점을 이해하지 못하면 계속해서 혼란이 발생한다.

특히,

셰이더가 안 먹히는 문제

포스트 프로세싱이 적용되지 않는 문제

라이팅 결과가 Built-in과 다른 문제

이런 현상들은 “URP라서 버그”가 아니라
다른 파이프라인을 사용하고 있기 때문에 발생하는 정상 동작이라는 점을 인지해야 한다.

다음에 같은 상황이 온다면:

프로젝트 생성 시 URP 여부를 먼저 확인하고

문제 발생 시 Built-in 기준 해결책을 그대로 적용하지 않고

반드시 URP 공식 문서를 기준으로 원인을 찾을 것이다.

한 줄 정리

URP는 Built-in의 옵션이 아니라, 렌더링 구조 자체가 다른 파이프라인이다.
